The Standards

WCAG 2.0

WCAG 2.0 is the second iteration of the standards written by the W3C to capture the functional parameters for what makes a website accessible.

While different sets of standards can exist across different national entities, the W3C, everyone's favorite standards writers, have assembled the Web Content Accessibility Guidlines, or WCAG. It's a collection of website qualities that ensure that your websites are usable and intelligible for folks regardless of what impairments they may or may not have.

What are the standards?

The overarching goal is to make sure that your pages are

  • Perceivable
  • Operable
  • Understandable
  • Robust
... In a nutshell, the goals are to ensure your site is perceivable — users must be able to perceive the information in some way — operable — all users must be able to operate the interface — understandable — users must be able to understand the information — and robust, being that the site can be used using accessed with a large number of user agents and future-proofed for the same.

The success criteria are then broken up across these four categories.

The success criteria are then broken up across these four categories, which are fine for groking the goals at a top level, but doesn't group them in a way that's easy to work into a project's lifecycle, so I went ahead and took a crack at it. Specific guidelines and references will be included throughout for later reference.

The Design Phase

The Design Phase: Typeface

The Design Phase: Typeface

Maximize readability for folks with low-vision and dyslexia by choosing a typeface with...

  • Longer ascenders and descenders to make different letterforms more distinct.
  • No ornamentation that obscures the letterforms.
  • Letterforms that are distinct from others that are formed in a similar way.

The Design Phase: Type Size

Embrace the relative! Set font-size, padding, and margin to relative units (rem, em, ch) to make it possible to zoom gracefully! (View WCAG SC 1.4.4)

Now that there are a variety of devices with varying pixel densities to consider, font-size can seem peskier than ever, but so long as you make sure to use relative units when setting your font-size, padding, and margin, your page will not look miserable when zooming out.

The Design Phase: Text

  • Keep fields of text aligned to the left.
  • For paragraphs, keep line-height: 1.5;
  • Bonus: Paragraphs between 45–75 characters long are most comfortable for reading, so keeping your paragraphs between that number of rem units is best.

(View WCAG SC 1.4.8)

The Design Phase: Color

When using color for text, be sure that the contrast ratio between the background and text is at least 4.5 for all text and at least 3 for text that's above 14pt and bold or above 18pt in size. (View WCAG SC 1.4.3)

The Design Phase: Images

If there's alt text, it will be read by screen readers.

  • INCLUDE when the image is a substitute for text
  • INCLUDE when a description of the image provides additional information
  • OMIT when the image only reinforces existing text.

The Development Phase

The Development Phase: Head First

Well, sort of.

  • Make sure you use the lang attribute in your html tag for proper reading and text rendering. (View WGAC SC 3.1.1)(View WGAC SC 3.1.2)
  • Use critical path CSS to prevent any background-images from causing load havoc.
- Note that you can use them for other elements that switch languages midstream. Difference between "Yo hab-luh espanole" and "Yo habla español" - Structuring your styles and scripts to load in a critical path fashion.

The Development Phase: Markup Order

As your design looks from the top down, so should your markup look. This ensures assistive technologies and keyboard users will be able to digest and navigate information in the same sequence as other users.

The Development Phase: Keeping Things Semantic

When possible, use semantically appropriate HTML tags for your content. HTML 5 does us a favor by offering us plenty to choose from.

- When possible, use semantically appropriate HTML tags for your content. This allows assistive technologies to be able to handle elements in the correct fashion without having to attach a myriad of aria attributes to an element and also to navigate properly within a webpage.

The Development Phase: Semantic Headers

Conveniently, <header> tags are for headers.

Attach the attribute of role="banner" to your primary header.

Conveniently, your header content is placed between header tags. And while it's possible to create confusion for yourself by having multiple headers in your markup, the role of banner plays by Highlander rules, which is to say, there can only be one.

The Development Phase: Semantic Navigation

Just like the with header, there are handy <nav> tags to serve this purpose.

Attach the attribute of role="navigation" for navigational, but non-<nav> elements.

Use the WAI-ARIA attribute of aria-label or reference a navigational area's header with aria-labelledby to make its purpose clear

Just like with the header, there are handy nav tags that serve the purpose of marking your navigational area for you. In cases where a page element serves a navigational purpose but isn't the primary navigation, you can use the role of navigation to it, which will flag it and announce it as such for screenreaders. For even better clarity, you can attach aria-label or aria-labelledby to your nav sections, which will ensure that screenreaders call out what you're looking at accordingly (for Wikipedia, these labels could result in two navigations being "Site Navigation" and "Article Contents")
