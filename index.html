The Standards

WCAG 2.0

WCAG 2.0 is the second iteration of the standards written by the W3C to outline elements of structure that make websites accessible.

While different sets of standards can exist across different national entities, the W3C, everyone's favorite standards writers, have assembled the Web Content Accessibility Guidlines, or WCAG. It's a collection of website qualities that ensure that your websites are usable and intelligible for folks regardless of what impairments they may or may not have.

What are the standards?

The overarching goal is to make sure that your pages are

  • Perceivable
  • Operable
  • Understandable
  • Robust
... In a nutshell, the goals are to ensure your site is perceivable — users must be able to perceive the information in some way — operable — all users must be able to operate the interface — understandable — users must be able to understand the information — and robust, being that the site can be used using accessed with a large number of user agents and future-proofed for the same. The success criteria are then broken up across these four categories, which are fine for groking the goals at a top level, but doesn't group them in a way that's easy to work into a project's lifecycle, so I went ahead and took a crack at it. Specific guidelines and references will be included throughout for later reference.

The Design Phase

Design: Typeface

Design: Typeface

Maximize readability for folks with low-vision and dyslexia by choosing a typeface with...

  • Longer ascenders and descenders to make different letterforms more distinct.
  • No ornamentation that obscures the letterforms.
  • Letterforms that are distinct from others that are formed in a similar way.

Design: Type Size

Embrace the relative! Set font-size, padding, and margin to relative units (rem, em, ch) to make it possible to zoom gracefully! (View WCAG SC 1.4.4)

Now that there are a variety of devices with varying pixel densities to consider, font-size can seem peskier than ever, but so long as you make sure to use relative units when setting your font-size, padding, and margin, your page will not look miserable when zooming out.

Design: Text

  • Keep fields of text aligned to the left.
  • For paragraphs, keep line-height: 1.5;
  • Bonus: Paragraphs between 45–75 characters long are most comfortable for reading, so keeping your paragraphs between that number of rem units is best.

(View WCAG SC 1.4.8)

Design: Textual Links

When possible, make it explicit what kind of content a link links to reduce cognitive load on the user. (View WCAG SC 2.4.9)

If for some reason, it can't be done, that's fine — so long as all users are left with a sense of mystery, this criteria still counts as passed.

Design: Textual Links

  • At the very least, make it clear where a given link might be sending a user based on context. (View WCAG SC 2.4.4)
  • To make this easier for assistive technologies, use the aria-label attribute to provide more information about where a link might take you or point at information that does describe a link using aria-labelledby

Design: Color

When using color for text, be sure that the contrast ratio between the background and text is at least 4.5 for all text and at least 3 for text that's above 14pt and bold or above 18pt in size. This should guarantee readability for users with 20/40 vision (View WCAG SC 1.4.3)

- 20/40 vision is around the average visual acuity for people around the age of 80.

Design: Color

To take it to the next level and be a hero to people with 20/80 vision, you can use a contrast ratio of 7 for your target, which should also resolve any readability issues for folks with varying forms of colorblindness. (View WCAG SC 1.4.6)

Design: Color

Any information imparted by charts or graphs by the use of color should be made available in another way for folks with varying types of colorblindness.(View WCAG 1.4.1)

Design: Images

If there's alt text, it will be read by screen readers.

  • INCLUDE when the image is a substitute for text
  • INCLUDE when a description of the image provides additional information
  • OMIT when the image only reinforces existing text.

Design: Images

What if connection speeds are dismal? What if your users are choosing to spend their mobile data on your site/app?

Design: Images

  • Compress your raster images! ImageOptim or Compressor.io can be handy here.
  • Selectively load images based on the device being used.
  • Use SVG where possible for iconography for small file sizes.

Design: Images of Text

  • Logos are permissible when augmented with alt text, but any other manifestations of image-based text should be strictly decorative. (View WGAC SC 1.4.5) (View WGAC SC 1.4.9)
  • Believe in yourself! You don't need an image for that type treatment! View some sweet typographic samples for inspiration.
    • You might ask yourself, "But what if I have an image that's of text? What then?" Unless it's a logotype, get it out of there! You are one slow network or loading failure away from your site being difficult from anyone to operate or depriving them of information. CSS is also powerful enough to make the dream work for everyone by making it possible to lay in custom typefaces or resize them without deferring to image use.

The Development Phase

Development: Head First

  • Make sure you use the lang attribute in your html tag for proper reading and text rendering. (View WGAC SC 3.1.1)(View WGAC SC 3.1.2)
  • Use <title> tags that describe the content at a glance. (View WCAG SC 2.4.2)
  • Use critical path CSS to prevent any background-images from causing load havoc.
  • FOUT (Flash of Unstyled Text) is superior to FOIT (Flash of Invisible Text)
- Note that you can use them for other elements that switch languages midstream. Difference between "Yo hab-luh espanole" and "Yo habla español" - Structuring your styles and scripts to load in a critical path fashion.

Development: Markup Order

As your page reads from the top down, so should your markup. This ensures assistive technologies and keyboard users will be able to digest and navigate information in the same sequence as other users. (View WCAG SC 2.4.3)

Development: Keeping It Semantic

When possible, use semantically-appropriate HTML tags for your content. HTML 5 does us a favor by offering us plenty to choose from.

- When possible, use semantically appropriate HTML tags for your content. This allows assistive technologies to be able to handle elements in the correct fashion without having to attach a myriad of aria attributes to an element and also to navigate properly within a webpage.

Development: Headers

Conveniently, <header> tags used to keep headers semantic.

Attach the attribute of role="banner" to your primary header.

Conveniently, your header content is placed between header tags. And while it's possible to create confusion for yourself by having multiple headers in your markup, the role of banner plays by Highlander rules, which is to say, there can only be one.

Development: Navigation

  • You don't have to reinvent the wheel — adhering to layout conventions common on the web allow you to piggyback on their existing familiarity.
  • Keep page navigation and controls in the same location as often as possible! This helps users familiarize themselves with the layout and anticipate where they'll find the content they seek in the future. (View WCAG SC 3.2.3)

Development: Navigation

Just like the with header, there are handy <nav> tags to serve this purpose.

Attach the attribute of role="navigation" for navigational, but non-<nav> elements.

Use the WAI-ARIA attribute of aria-label or reference a navigational area's header with aria-labelledby to make its purpose clear

Just like with the header, there are handy nav tags that serve the purpose of marking your navigational area for you. In cases where a page element serves a navigational purpose but isn't the primary navigation, you can use the role of navigation to it, which will flag it and announce it as such for screenreaders. For even better clarity, you can attach aria-label or aria-labelledby to your nav sections, which will ensure that screenreaders call out what you're looking at accordingly (for Wikipedia, these labels could result in two navigations being "Site Navigation" and "Article Contents")

Development: Navigation

Making search functionality available also provides a nice bump to accessibility, as well as providing opportunities to navigate across pages in normal flow outside of the main navigational fixtures. (View WCAG SC 2.4.5)

Development: Navigation

When possible, keep it clear where the user is currently located, whether providing a visual indicator within the site's navigation or providing breadcrumbs marking their path. (View WCAG SC 2.4.8)

Development: Navigation

When utilizing dropdown or slide-out navigation, use role and aria attributes to prevent a complete readout of submenus and keep from "losing" focus. (View WCAG SC 2.4.1)

Development: Navigation

Speaking of :focus, leave that blue halo serving as a focus indicator alone unless you're replacing it with something else! (View WCAG SC 2.4.7)

This is the only way someone using a keyboard can keep their bearings as to where they are on a page, so do them a solid and leave it be!

Development: Navigation

To prevent a full read-out of navigation or forcing a keyboard user to tab through it again on a page load, provide a "Skip to Content Link" in your header before the <nav> that's only visible to the user on :focus.

Development: Main

Use the <main> tags for your primary content area, as well as the role="main" landmark as an attribute.

When linking down a page, add tabindex="-1" as an attribute to sections targeted by links to allow the focus area to jump accordingly.

Development: Headings

When content is broken into sections, be sure to make headings as descriptive of the content as possible and use them in sequence (i.e. h1..h6) to make content more digestable and navigable. (View WCAG SC 2.4.10)

Development: Forms

When constructing forms, be sure that all of your form fields have labels that they're associated to. (View WCAG SC 2.4.6)

Development: Forms

Don't depend on placeholder text as a label replacement. It can provide context as to what belongs in a form field, but once a user begins to type in it, that context is lost and unrecoverable for purposes of form review.

Development: Communicating State

  • Mark elements that are hidden within the interface with the aria-hidden="true" attribute and toggle its state to false with JavaScript when it changes.
Screen readers read markup, not what's actively "visible" on the page. so

Development: Communicating State

  • For users to keep the best grasp on context, make redirects, where they occur, as instantaneous as possible.
  • Only allow the page to refresh upon being triggered by the user to keep page transitions expected (View WCAG SC 3.2.5)

Development: Screenreaders and "Secret" Text

For non-sighted users who need additional context provided for other users via images or CSS, provide content that will be picked up by screen readers but "invisible" to other users with this handy dandy utility class:

.visually-hidden { // For text we want read out for screen readers, but not visible generically.
  position: absolute;
  width: 1px;
  height: 1px;
  overflow: hidden;
  clip: rect(1px, 1px, 1px, 1px);
}
